Container-demuxer rebuild — DTS re-enabled for mp4/ts/m2ts (webOS 25 / C5)
==========================================================================

Source: LG webOS-25 GStreamer 1.24 monorepo (lgstreamer/gstreamer-webos-25).
Built reproducibly by build-demux.sh (debian:11-slim, arm-linux-gnueabi-gcc, meson).

Why a rebuild was needed
------------------------
LG ships isomp4/mpegtsdemux with the DTS caps paths compiled out. Building with
-Ddca=true recompiles those paths, but they are ALSO gated at runtime by a
GObject property `dts_support` that LG defaults to FALSE — so mp4/ts DTS still
fell out as untargetable audio/x-gst-fourcc-dtsc. The fix is a 2-line source
patch flipping that default to TRUE:

  gst-plugins-good/gst/isomp4/qtdemux.c        qtdemux->dts_support = FALSE -> TRUE
  gst-plugins-bad/gst/mpegtsdemux/tsdemux.c    demux->dts_support   = FALSE -> TRUE

(each file has exactly one such assignment; nothing else changed; no later code
re-sets it to FALSE.)

ABI (both artifacts, stripped) — must match the C5 GStreamer userspace
---------------------------------------------------------------------
  file        : ELF 32-bit LSB shared object, ARM, EABI5 (soft-float)
  e_flags     : 0x05000200   (od -An -tx4 -j36 -N4)
  interpreter : /lib/ld-linux.so.3
  max GLIBC   : 2.7  (well under the C5's 2.35 ceiling)
  GStreamer   : 1.24
  strings     : audio/x-dts + "DTS audio" present

On-device validation (real LG C5, via the media registry)
---------------------------------------------------------
Real Blu-ray DTS-HD MA 5.1 in .ts  -> gst-discoverer reports "audio: DTS,
  6 channels (FL FR FC LFE RL RR), 48000 Hz, 32-bit" and decodes to non-silent
  S32LE PCM end-to-end (dtsdec).
mp4 (dtsc)                          -> decodes to PCM.
normal AAC mp4                      -> unaffected (no regression).
(A residual/XLL-only DTS-HD test m2ts with no decodable core stays silent — a
 content limitation; reference ffmpeg fails on it too. Demux/routing is correct.)
